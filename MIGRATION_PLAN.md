# План миграции: гибрид 2GIS + kudikina.ru

## Суть

2GIS остаётся основой: геокодинг, маршрутизация, пешая навигация, структура маршрута.
Kudikina.ru подменяет только расписание автобусов — точнее и полнее.

## Как работает склейка

```
Пользователь: /route «Дом → Работа»
         │
         ▼
    ┌─ 2GIS Routing API ─────────────────────────────┐
    │  Структура маршрута:                            │
    │  📍 пешком 390 м (5 мин)                        │
    │  🚌 остановка «По требованию» → автобус 24      │
    │     → «Студенческая» (9 остановок)              │
    │  🚶 пешком 210 м (3 мин)                        │
    │  🏁 финиш                                       │
    │                                                 │
    │  schedules: [06:43, 06:58, ...]  ← НЕТОЧНЫЕ     │
    └─────────────────────────────────────────────────┘
         │
         │ Извлекаем: start_stop="По требованию"
         │            end_stop="Студенческая" (из platforms)
         │            bus_number="24"
         ▼
    ┌─ kudikina.ru ───────────────────────────────────┐
    │  GET /omsk/search?a=По+требованию&b=Студенческая│
    │                                                 │
    │  Автобус 24: 06:15, 06:31, 06:43R, 06:53R,     │
    │  06:58, 07:03R, ... (полный день)   ← ТОЧНЫЕ    │
    └─────────────────────────────────────────────────┘
         │
         │ Совпадение по номеру "24" → подменяем schedules
         ▼
    ┌─ Результат ─────────────────────────────────────┐
    │  🕐 Выход 6:38 → посадка 6:43 → прибытие ~7:05 │
    │  🚏 Расписание: 6:43R, 6:53R, 6:58, 7:03R, ... │  ← из kudikina
    │  📏 4200 м | 🚶 600 м                            │  ← из 2GIS
    │  🚌 По требованию → автобус 24                   │  ← из 2GIS
    │     Остановки: По требованию → ... → Студенческая│  ← из 2GIS
    └─────────────────────────────────────────────────┘
```

## Ключевая функция: `enrich_with_kudikina()`

Одна функция-обогатитель, которая встраивается между получением маршрутов
от 2GIS и отображением/уведомлениями:

```python
async def enrich_with_kudikina(route: RouteInfo, city_slug: str = "omsk") -> RouteInfo:
    """Подменить расписание 2GIS на точное расписание kudikina."""

    # 1. Извлечь из 2GIS movements:
    #    - start_stop (waypoint.name первого passage)
    #    - end_stop (последний элемент platforms.names последнего passage)
    #    - bus_number (routes[].names[] из passage)

    # 2. Запросить kudikina:
    #    kudikina_search(city_slug, start_stop, end_stop)

    # 3. Найти совпадение по номеру автобуса (нечёткое: "24" == "24")

    # 4. Конвертировать kudikina times → формат 2GIS schedules:
    #    [{"type": "precise", "precise_time": "6:43", "start_time": 24180}, ...]

    # 5. route.schedules = new_schedules
    #    → все существующие методы (all_schedule_minutes, _calc_times_from_schedule,
    #      _format_schedule_list, format_summary) работают без изменений
```

## Что меняется в каждом файле

### kudikina_client.py (новое)
- Добавить `schedule_minutes()` в `KudikinaRoute` — список времён в минутах
  (аналог `RouteInfo.all_schedule_minutes()`)

### models.py (новое)
- Добавить метод `extract_passage_info()` в `RouteInfo` — извлечение
  start_stop, end_stop, bus_number из movements
- Добавить `enrich_with_kudikina()` — функция-обогатитель

### handlers.py (минимальные изменения)
- `/route`: после `fetch_routes()` → вызвать `enrich_with_kudikina()` для каждого route
- Всё остальное (format_summary, фильтрация) — без изменений

### scheduler.py (минимальные изменения)
- `_fetch_and_filter()`: после получения routes → вызвать `enrich_with_kudikina()`
- Всё остальное (_check_trip_notifications, _check_go_notification) — без изменений

### Не меняются
- `api_client.py` — как было
- `geocoder.py` — как было
- `config.py` — добавить `DEFAULT_CITY_SLUG`
- `bot.py` — без изменений
- `storage.py` — без изменений
- `Trip` модель — без изменений (!)

## Порядок миграции

### Этап 1: Функция обогащения (~1 сессия)
1. Добавить `extract_passage_info()` в RouteInfo
2. Добавить `schedule_minutes()` в KudikinaRoute
3. Написать `enrich_with_kudikina()` с матчингом по номеру
4. Тест: запуск на примере, сравнение schedules до/после

### Этап 2: Интеграция в /route (~10 мин)
1. В `cb_route()`: после fetch_routes → enrich каждый route
2. Проверить вывод format_summary

### Этап 3: Интеграция в уведомления (~10 мин)
1. В `_fetch_and_filter()`: после получения routes → enrich
2. Проверить all_schedule_minutes возвращает kudikina-данные

### Этап 4: Обработка edge cases
1. Kudikina недоступна → fallback: оставляем 2GIS schedules (graceful degradation)
2. Автобус не найден на kudikina → оставляем 2GIS schedules
3. Маршруты с пересадками → enrich каждый passage-сегмент отдельно

## Сравнение подходов

| | Полная замена (старый план) | Гибрид-overlay (новый план) |
|---|---|---|
| Изменений в файлах | 8+ файлов | 3 файла |
| Trip модель | Новые обязательные поля | Без изменений |
| /add флоу | +2 FSM состояния | Без изменений |
| /setstop | Удалить/переделать | Без изменений |
| /edit | Переделать | Без изменений |
| Пешая навигация | Теряется | Сохраняется |
| Расчёт времён | Переписать | Работает как есть |
| Fallback при ошибке | Нет (всё на kudikina) | Есть (2GIS schedules) |
| Точность расписания | kudikina | kudikina |

## Риски (те же, но менее критичны)

1. **kudikina.ru недоступен** → graceful degradation: показываем 2GIS расписание.
   Пользователь не заметит ошибки, просто данные менее точные.

2. **HTML-парсинг хрупкий** → тот же graceful degradation.

3. **Несовпадение номеров** — 2GIS может назвать "24", kudikina "24Н" или наоборот.
   Решение: нормализация номеров, fuzzy matching.

4. **Несовпадение названий остановок** — 2GIS "3-я Енисейская", kudikina
   "Улица 3-я Енисейская". Решение: поиск подстроки при матчинге.
